# CLAUDE.md

이 문서는 **day4 프로젝트에서 코드를 작성할 때 반드시 따라야 하는 개발 가이드**다.
"무엇을 만들 것인가"는 `game-designer` 서브 에이전트(`.claude/agents/game-designer.md`)가 작성하는 PRD를 따른다.
이 문서는 "어떻게 만들 것인가"만 다룬다. 게임 컨셉, 기능 설명, 일정 등 PRD에 속하는 내용은 여기에 쓰지 않는다.

## 1. 기술 스택

- **언어**: TypeScript (strict 모드 필수, `tsconfig.json`의 `strict: true` 유지)
- **빌드 도구**: Vite
- **런타임 대상**: 브라우저 (데스크톱/모바일 웹)
- **게임 렌더링/루프**: Phaser 3 (2D 캔버스 기반 게임 엔진). 게임 로직·씬 관리가 필요 없는 단순 UI(메뉴, 설정 화면 등)는 React로 구현하고, 실제 게임 플레이 화면은 Phaser가 소유한다.
- **상태 관리**: 전역 상태가 필요한 경우 Zustand 사용. Redux 등 무거운 상태관리 라이브러리는 도입하지 않는다.
- **패키지 매니저**: npm
- **테스트**: Vitest (게임 로직/시스템 단위 테스트), Playwright는 필요 시에만 도입
- **린트/포맷**: ESLint + Prettier (아래 코드 스타일 규칙을 강제하는 룰셋 사용)

> 프레임워크를 변경해야 할 특별한 이유가 없다면 위 스택을 임의로 바꾸지 않는다. 바꿔야 한다면 왜 바꾸는지 사용자에게 먼저 확인한다.

## 2. 코드 스타일

- **네이밍**
  - 변수/함수: `camelCase`
  - 클래스/React 컴포넌트/Phaser Scene: `PascalCase`
  - 상수(게임 밸런스 수치 등): `SCREAMING_SNAKE_CASE`
  - 파일명: `kebab-case` (컴포넌트 파일은 예외적으로 `PascalCase.tsx` 허용)
- **타입**
  - `any` 금지. 불가피하면 `unknown` + 타입 가드로 좁힌다.
  - 게임 엔티티, 이벤트, 밸런스 데이터는 반드시 인터페이스/타입으로 정의한다.
- **폴더 구조** (예시, 프로젝트 초기화 시 이 구조를 기준으로 시작)
  ```
  src/
    scenes/      # Phaser Scene (화면 단위)
    entities/    # 플레이어, 몬스터 등 게임 오브젝트 클래스
    systems/     # 전투, 인벤토리 등 규칙/로직 (렌더링과 분리)
    ui/          # React로 만든 메뉴/HUD 등 UI 컴포넌트
    config/      # 밸런스 수치, 상수, JSON 데이터
    assets/      # 이미지, 사운드, 스프라이트시트
  ```
- **주석**: 기본적으로 주석을 쓰지 않는다. 코드만 봐서는 알 수 없는 이유(밸런스 수치를 특정 값으로 정한 이유, 엔진의 알려진 버그를 피하기 위한 우회 코드 등)가 있을 때만 한 줄로 남긴다.
- **포맷팅**: Prettier 기본 설정 사용, 세미콜론 사용, 싱글 쿼트.

## 3. 개발 원칙

- **로직과 렌더링 분리**: 게임 규칙(데미지 계산, 쿨다운, 인벤토리 등)은 `systems/`에 순수 함수/클래스로 작성하고 Phaser Scene은 그 결과를 그리기만 한다. 이렇게 해야 렌더링 없이도 게임 로직을 단위 테스트할 수 있다.
- **데이터 주도 설계**: 데미지, 쿨다운, 드랍률 같은 밸런스 수치는 코드에 하드코딩하지 않고 `config/`의 상수나 JSON으로 분리한다. 기획 변경 시 코드 수정 없이 값만 바꿔서 반영할 수 있어야 한다.
- **PRD 추적성**: 새로운 기능을 추가하기 전에 해당 기능이 PRD의 어느 섹션에 대응하는지 확인한다. PRD에 없는 기능을 임의로 추가하지 않는다(스코프 확장은 먼저 사용자에게 확인).
- **과설계 금지 (YAGNI)**: 지금 필요하지 않은 확장성을 위한 추상화, 플러그인 구조, 설정 옵션을 미리 만들지 않는다. PRD의 "개발 우선순위"에서 P0/MVP로 분류된 범위부터 구현한다.
- **성능**: 게임 루프(매 프레임 실행되는 코드)에서 객체를 새로 생성하지 않는다. 총알, 파티클처럼 자주 생성/삭제되는 오브젝트는 오브젝트 풀링을 사용한다. 목표 프레임레이트는 60fps.
- **테스트**: `systems/`의 게임 로직은 Vitest로 단위 테스트를 작성한다. 렌더링/입력 반응 등 눈으로 확인해야 하는 부분은 브라우저에서 직접 플레이해 검증한다(타입 체크와 유닛 테스트 통과만으로 "기능이 실제로 재미있게 동작한다"를 증명하지 않는다).
- **에러 처리**: 실제로 발생 가능한 상황(네트워크 저장 실패, 잘못된 세이브 데이터 등)에만 에러 처리를 추가한다. 일어날 수 없는 케이스에 대한 방어 코드는 작성하지 않는다.
